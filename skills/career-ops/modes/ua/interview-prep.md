# Режим: interview-prep — Підготовка до співбесіди

Коли користувач просить підготуватися до співбесіди в конкретній компанії, або оцінка ≥ 4.0 і статус оновлено на `Interview`.

## Вхідні дані

1. **Назва компанії** та **посада** (обов'язково)
2. **Звіт оцінки** в `reports/` (якщо є)
3. **Банк історій** `interview-prep/story-bank.md`
4. **CV** `cv.md` + `article-digest.md`
5. **Профіль** `config/profile.yml` + `modes/_profile.md`

## Крок 1 — Дослідження

WebSearch-запити. Витягати структуровані дані, не перекази. Цитувати джерела.

| Запит                                                       | Що витягти                                               |
| ----------------------------------------------------------- | -------------------------------------------------------- |
| `"{company} {role} interview questions site:glassdoor.com"` | Реальні питання, складність, таймлайн, кількість раундів |
| `"{company} interview process site:teamblind.com"`          | Описи процесу, дані про компенсацію                      |
| `"{company} {role} interview site:leetcode.com/discuss"`    | Конкретні задачі, теми system design                     |
| `"{company} engineering blog"`                              | Стек, цінності, технічні пріоритети                      |
| `"{company} співбесіда відгуки site:dou.ua"`                | Відгуки українською про процес співбесіди                |
| `"{company} інтерв'ю досвід site:dou.ua"`                   | Досвід проходження співбесід на DOU                      |
| `"{company} відгуки співробітників site:dou.ua"`            | Відгуки про компанію від працівників                     |
| `"{company} interview site:djinni.co"`                      | Відгуки кандидатів на Djinni                             |

**НЕ вигадувати питання.** Виведені з JD — позначати `[виведено з JD]`.

## Крок 2 — Огляд процесу

```markdown
## Огляд процесу

- **Раунди:** {N} раундів, ~{X} днів від початку до кінця
- **Формат:** {напр., скринінг HR → технічне інтерв'ю → тестове → onsite → наймаючий менеджер}
- **Складність:** {X}/5 (середнє Glassdoor, N відгуків)
- **Позитивний досвід:** {X}%
- **Особливості:** {напр., "парне програмування замість дошки", "без LeetCode, тільки практичні задачі"}
- **Джерела:** {посилання}
```

## Крок 3 — Розбір за раундами

```markdown
### Раунд {N}: {Тип}

- **Тривалість:** {X} хв
- **Проводить:** {колега / менеджер / skip-level / рекрутер}
- **Що оцінюють:** {конкретні навички}
- **Зафіксовані питання:**
  - {питання} — [джерело: Glassdoor 2026-Q1]
- **Як готуватися:** {1-2 конкретні дії}
```

## Крок 4 — Вірогідні питання

### Технічні

Питання з system design, написання коду, архітектури, domain knowledge.

### Поведінкові

Питання про лідерство, конфлікти, колаборацію, невдачі.
Для кожного — яка історія з `story-bank.md` підходить найкраще.

### Специфічні для ролі

Питання, прив'язані до конкретного JD (з урахуванням архетипу).

### Red flags у бекграунді

Питання, які інтерв'юер вірогідно задасть про прогалини, переходи або незвичайні елементи у бекграунді. Читати `_profile.md` та `cv.md`.

## Крок 5 — Маппінг банку історій

| #   | Вірогідне питання/тема | Найкраща історія з story-bank.md | Fit | Прогалина? |
| --- | ---------------------- | -------------------------------- | --- | ---------- |

- **strong**: історія прямо відповідає на питання
- **partial**: історія суміжна, потрібний рефреймінг
- **none**: немає підходящої → відзначити для користувача

## Крок 6 — Чекліст технічної підготовки

```markdown
- [ ] {тема} — чому: "{доказ з дослідження}"
- [ ] {тема} — чому: "{їхній блог/продукт каже, що це важливо}"
```

Пріоритизувати за частотою та релевантністю. Максимум 10 пунктів.

## Крок 7 — Сигнали компанії

- **Цінності, на які скрінять:** назвати, цитувати джерело
- **Словник для використання:** внутрішні терміни компанії
- **Чого уникати:** анти-патерни з відгуків
- **Питання їм:** 2-3 гострих питання, що демонструють домашню роботу

## Специфіка українських співбесід 🇺🇦

### Типові раунди в українських ІТ-компаніях

1. **Скринінг HR** (20-30 хв): мотивація, зарплатні очікування, формат роботи, англійська
2. **Технічне інтерв'ю** (60-90 хв): знання стеку, архітектура, практичні задачі
3. **Live coding** (45-60 хв): алгоритми або практичні задачі (частіше практичні)
4. **System design** (60 хв): проєктування систем (особливо у продуктових компаніях)
5. **Культурний fit** (30-45 хв): цінності, командна робота, комунікація
6. **Тестове завдання** — поширене в українських компаніях (1-4 години, іноді take-home)

### Часті питання на українських співбесідах

- "Розкажіть про себе" (2-3 хвилини, структуровано)
- "Чому шукаєте нову роботу?"
- "Зарплатні очікування?" (уточнити USD/UAH, ФОП/штат)
- "Готовність до переїзду / timezone overlap?"
- "Розкажіть про найскладніший проєкт"
- "Як вирішуєте конфлікти в команді?"
- "Який ваш рівень англійської?" (часто перевіряють в процесі)
- System design: "Спроєктуйте {URL shortener / чат / стрічка новин / платіжна система}"

### Формат STAR+R українською

- **С (Ситуація):** Контекст і вступні дані
- **З (Завдання):** Що потрібно було зробити
- **Д (Дії):** Що конкретно зроблено
- **Р (Результат):** Вимірюваний результат
- **О (Осмислення):** Що зрозумів, що зробив би інакше

## Висновок

Зберегти звіт у `interview-prep/{company-slug}-{role-slug}.md`.

```markdown
# Підготовка до співбесіди: {Компанія} — {Роль}

**Звіт:** {посилання на звіт оцінки або "N/A"}
**Досліджено:** {YYYY-MM-DD}
**Джерела:** {N} відгуків Glassdoor, {N} постів Blind, {N} інших
```

## Правила

- **НІКОЛИ не вигадувати питання і приписувати їх джерелам.**
- **НІКОЛИ не фабрикувати рейтинги Glassdoor.**
- **Цитувати все.** Кожне питання, кожна статистика отримує джерело або тег `[виведено]`.
- Генерувати мовою JD (UA за замовчуванням для україномовних вакансій).
- Бути прямим. Це робочий документ підготовки, не мотиваційна промова.
