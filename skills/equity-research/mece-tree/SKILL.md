# MECE 问题树拆解

## 触发条件

用户提出一个模糊或复杂的商业问题，例如：
- "留存率下降了8%"
- "Q3毛利率为什么下滑"
- "新业务该不该做"
- "帮我拆一下这个问题的分析框架"

## 输出结构

### Step 1：确认核心问题

将用户的问题精炼为一句可被验证的陈述。如果不能，先和用户对齐。

### Step 2：MECE 拆解（3-5个一级分支）

原则：
- **Mutually Exclusive** — 分支之间不重叠
- **Collectively Exhaustive** — 合起来覆盖全部可能

常见拆解维度：
- **结构拆解**：收入 = 客单价 × 客户数 × 复购率
- **流程拆解**：获客 → 激活 → 留存 → 变现 → 传播
- **内外拆解**：内部因素（产品/团队/运营）vs 外部因素（市场/竞争/政策）
- **时间拆解**：短期/中期/长期

### Step 3：可验证假设

每个叶子节点配套一个具体假设，格式：
- **假设**：XXX 是主要原因
- **验证方式**：用什么样的数据可以证实或证伪
- **预期范围**：如果假设成立，数据应该在什么范围

### Step 4：数据采集清单

输出一个可执行的数据清单，包含：
- 需要什么数据
- 数据来源（内部系统/公开数据/访谈获取）
- 优先级（P0 必须 / P1 建议）

## 输出格式

```
🔍 核心问题：[精炼后的问题]

📊 MECE 拆解
一级分支1
  ├─ 二级分支1.1
  │   └─ 假设：XXX → 验证方式：XXX
  ├─ 二级分支1.2
  └─ ...

一级分支2
  ├─ 二级分支2.1
  └─ ...

📋 数据采集清单
| 优先级 | 数据项 | 来源 | 用途 |
|--------|--------|------|------|
| P0 | XXX | XXX | 验证假设1.1 |
| ... | ... | ... | ... |

💡 下一步建议
```

## 注意事项

- 不要为了凑 MECE 把简单问题复杂化。如果问题只需 2 个分支就覆盖了，就是 2 个。
- 假设必须有可证伪性。"用户满意度下降"不是假设（无法直接验证），"NPS 评分从 45 降至 32"才是。
- 数据采集清单要能直接执行，不要提"需要用户画像数据"这种模糊的东西。
